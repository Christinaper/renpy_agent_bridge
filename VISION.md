# VISION: LLM-Enhanced Multi-Modal Access

## 核心愿景

游戏不应该只把状态藏在画面里。

如果一个游戏的真实状态、可交互对象、当前合法动作、叙事文本和视觉/音频语义只存在于渲染结果中，辅助技术就只能在外部猜测：OCR、图像识别、坐标点击、窗口自动化。

这个项目探索另一条路：

> 游戏运行时原生暴露开发者标注的语义真相，LLM 或其他适配器把这些语义转换为多种人类友好的体验。

## JSO
JSON 不是最终用户界面。

JSON 是中间层：

```text
developer-authored semantic truth
  -> JSON bridge
  -> LLM / adapter
  -> human-facing output
```

所以批评“JSON 不如画面适合人类”并不命中目标。人类不需要直接阅读 JSON。JSON 是给机器适配器消费的。

## LLM 的角色

LLM 不是游戏本体，也不是 A11y 本体。

LLM 是适配器层：

- 把语义状态转成自然语言讲述。
- 把复杂事件转成简短摘要。
- 按用户偏好调整描述粒度。
- 把玩家意图转成结构化动作。
- 为非视觉体验生成连续叙事。
- 为测试或研究生成可读日志。

同一份语义可以被转换成很多输出：

- 详细语音描述
- 一句话摘要
- 屏幕阅读器友好的文本
- 命令行式交互
- 认知辅助提示
- 自动测试动作
- 剧情回放日志

## 开发者语义真相

关键不是让模型从画面猜。

关键是开发者在游戏里写入语义：

- 当前场景是什么
- 视觉上发生了什么
- 音频提示代表什么
- 玩家现在能做什么
- 每个动作的语义是什么
- 当前选择有什么认知负荷
- 玩家状态如何变化

这些信息由游戏运行时导出，LLM 只负责适配和表达。

## 与 self-voicing 的关系

这个项目不是要替代 Ren'Py self-voicing。

self-voicing 是直接面向玩家的现有能力。这个项目关注的是更底层的语义接口：

- self-voicing 可以成为一个输出客户端。
- LLM 可以成为另一个输出客户端。
- 自动测试可以成为第三个客户端。
- 未来 screen reader bridge 也可以成为客户端。

目标不是减少体验形式，而是让同一份语义支持更多体验形式。

## 与 WCAG 的关系

v0.2 不声称符合 WCAG，也不尝试建立完整无障碍标准。

当前阶段只是证明：

```text
游戏运行时 -> 语义 JSON -> 外部适配器 -> 结构化动作
```

这条通路成立。

完整标准、可用性研究、screen reader 集成、键盘/开关设备体验、认知辅助规范，都应在后续阶段逐步验证。

## 服务对象

这条路线不只服务视障玩家。

它也可能服务：

- 想闭眼听剧情的人
- 想用语音玩游戏的人
- 想要低视觉负担体验的人
- 想要认知辅助和摘要的人
- 想要自动测试剧情分支的开发者
- 想研究叙事决策的研究者
- 想把游戏内容转成多模态体验的创作者

## 阶段路线

### v0.2 Bridge Prototype

Ren'Py 导出 JSON，Agent 读写文件，证明闭环。

### v0.3 Runtime Health

让桥接运行时更稳定：session-safe、non-blocking、observable、failure-aware。

### v0.4 Semantic Depth

扩展语义表达：对象、视觉事件、音频事件、动作后果、认知负荷、上下文历史。

### v0.5 Assistive Clients

探索多个客户端：LLM narration、screen reader bridge、keyboard-only client、test bot。

### v0.6 Authoring Workflow

让游戏作者能低成本标注和验证语义。

## 北极星

长期目标不是让某个模型“会玩游戏”。

长期目标是：

> 游戏把可访问语义作为运行时基础设施暴露出来，LLM 和其他适配器把同一份语义转换成不同玩家、开发者和研究者需要的体验。
