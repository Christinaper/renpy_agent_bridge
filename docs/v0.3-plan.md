# v0.3 Plan Handoff: Runtime Health

## 阶段定位

v0.3 的主题是：

```text
Runtime Health
```

它不是完整 A11y 标准阶段，也不是 LLM 智能性阶段。

v0.3 的目标是把 v0.2 已经验证过的桥接闭环，从“能证明”推进到“更稳定、更可观察、更可重复验证”。

## 背景

v0.2 已验证：

```text
Ren'Py -> state.json -> Agent/LLM -> action.json -> Ren'Py -> finished
```

但 v0.2 仍有运行时问题：

- 旧 `state.json` 会污染新一轮运行。
- Ren'Py 等待 Agent 时 UI 会假死。
- Agent 失败时缺少清晰错误状态。
- 当前流程对启动顺序敏感。
- 调试主要依赖终端输出，不够可观察。

这些问题会妨碍后续验证，但不需要扩展语义模型才能解决。

## v0.3 核心目标

### 1. Session Isolation

目标：

- 每次游戏运行有唯一 `session_id`。
- `state.json` 和 `action.json` 都携带 `session_id`。
- Agent 只响应当前 session。
- 旧的 `finished` state 不会让 Agent 提前退出。

验收标准：

```text
Given stale exports from a previous run,
When a new Ren'Py session starts,
Then the agent ignores stale state and responds only to the active session.
```

### 2. Non-blocking Wait

目标：

- Ren'Py 等待 Agent 时 UI 不再假死。
- 移除主线程中的阻塞式 `time.sleep()` 等待。
- 使用 Ren'Py 友好的 `pause` / timer / screen loop 机制。

验收标准：

```text
Given the game is waiting for an agent action,
When the agent has not responded yet,
Then the Ren'Py window remains responsive.
```

### 3. Debug Observability

目标：

- 游戏 UI 或 debug overlay 显示桥接状态。
- 至少显示：
  - `session_id`
  - `turn_id`
  - `mode`
  - 当前文本
  - 可用 actions

验收标准：

```text
Given the bridge is awaiting action,
When the developer looks at the game window,
Then they can see which turn and actions are active.
```

### 4. Timeout / Error Mode

目标：

- Agent 未响应、action 格式错误、session 不匹配、turn 不匹配时，不再静默卡住。
- Ren'Py 导出结构化错误状态。

示例：

```json
{
  "mode": "error",
  "error": {
    "code": "agent_timeout",
    "message": "No action received before timeout"
  }
}
```

验收标准：

```text
Given the agent is absent,
When timeout is reached,
Then state.json contains mode=error and a structured error object.
```

### 5. Basic Agent Fallback

目标：

- 单一 `advance` 仍不调用 LLM。
- Ollama 超时可重试。
- LLM 输出无法解析时 fallback 到合法动作。
- fallback 需要记录原因。

验收标准：

```text
Given Ollama returns invalid output,
When a choice action is required,
Then the agent chooses a valid fallback action and logs the fallback reason.
```

## v0.3 不做什么

为了保持阶段边界，v0.3 不做：

- 不扩展到 Unity / Godot。
- 不做完整 screen reader 客户端。
- 不做 WCAG 合规声明。
- 不替代 Ren'Py self-voicing。
- 不做完整对象模型。
- 不做动作后果预测。
- 不追求 LLM 决策质量。
- 不把文件轮询立即替换成 WebSocket 或 HTTP API。

## 与 A11y 路线的关系

v0.3 仍然服务 infra 层 A11y：

- 不是让外部工具更会看图。
- 是让游戏运行时更可靠地暴露语义。
- 是让 LLM/adapter 有稳定输入和输出协议。

Runtime Health 是后续 Semantic Depth 和 Assistive Clients 的地基。

## 建议任务拆分

### Task 1: Spec Draft

只改文档和 schema 草案：

- 增加 `session_id`
- 增加 `mode: "error"`
- 增加 `error` object
- 定义 action/session 校验规则

### Task 2: Session ID

实现 session 隔离：

- Ren'Py 生成 session
- state/action 携带 session
- Agent 校验 session

### Task 3: Non-blocking Wait

替换阻塞等待：

- 移除 Ren'Py 主线程 `time.sleep()`
- 使用 `pause` 或 screen/timer

### Task 4: Error Mode

实现错误状态：

- timeout
- malformed action
- wrong session
- wrong turn

### Task 5: Debug Overlay

增加开发者可见状态：

- turn
- mode
- actions
- last error

### Task 6: Agent Fallback

增加 Agent 鲁棒性：

- Ollama retry
- parse fallback
- structured log

## v0.3 验证方式

建议新增：

```text
docs/v0.3-validation.md
docs/v0.3-validation.zh.md
```

验证用例：

1. 旧 exports 存在时启动新游戏。
2. Agent 晚于游戏启动。
3. Agent 早于游戏启动。
4. Agent 不启动，Ren'Py 进入 timeout/error。
5. action.json session_id 错误。
6. action.json turn_id 错误。
7. Ollama 返回无法解析文本。
8. 完整 16 turn 回归测试仍通过。

## 成功定义

v0.3 成功时，应能写下：

```text
The v0.2 semantic JSON bridge is now session-safe, non-blocking, observable, and failure-aware.
```

这时项目才适合进入 v0.4：Semantic Depth。
