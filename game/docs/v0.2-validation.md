# v0.2 Validation: Semantic JSON Loop Proof

## Hypothesis

> A visual novel runtime can expose narrative state and available player actions as semantic JSON, allowing an external agent to complete a playthrough without pixel-level input.

This validation is about the bridge loop, not about proving that the LLM is intelligent or that the project is production-ready.

## Result

**Date:** 2026-05-15
**Test game:** A11y-RenPy-Bridge demo
**Engine:** Ren'Py
**Reference agent:** `agent/simple_player.py`
**LLM runtime:** Ollama
**Model:** `llama3.2:1b`

The v0.2 semantic loop completed successfully.

## What Worked

1. `state.json` carried enough semantic information for the agent to act.
2. `turn_id` synchronized each state/action pair.
3. `mode` allowed the agent to distinguish active play from completion.
4. `advance` represented ordinary dialogue progression.
5. `choice` represented menu decisions.
6. `action.json` successfully drove Ren'Py forward.
7. The game completed without screen vision or coordinate-based automation.

## Accepted v0.2 Limitations

These are accepted limitations for the bridge prototype:

1. Blocking wait can make the Ren'Py UI appear frozen.
2. A clean run requires clearing stale `game/exports/*` files.
3. There is no `session_id`.
4. There is no robust retry/fallback for Ollama timeout or malformed output.
5. Choice turns currently reuse the previous `narrative.current_text` as context.

The fifth item is a data-quality issue, but it is not a v0.2 blocker. It should be addressed in v0.3 with explicit choice context.

## Validation Conclusion

v0.2 validates the bridge prototype:

```text
Semantic JSON Loop Proof Validated
```

The result supports the project direction: accessibility should be implemented through runtime-owned semantic interfaces, not only through external visual interpretation.

## Suggested Next Validation

v0.3 should validate runtime health:

- Agent can start before or after the game without reading stale state.
- Ren'Py UI remains responsive during waits.
- Session identity prevents cross-run contamination.
- Errors produce observable states instead of silent blocking.
- Choice context is represented explicitly.
